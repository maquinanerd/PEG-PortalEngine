## OBJETIVO PRINCIPAL

Implementar no PEG Portal Engine o suporte a **Site Profiles JSON**.

Um Site Profile é um arquivo JSON único por portal, contendo todas as configurações necessárias para transformar um WordPress recém-instalado em portal editorial.

Exemplo de uso desejado:

1. Eu crio um arquivo em `config/sites/aeconomia.json`.
2. Abro o painel do PEG Portal Engine.
3. Seleciono esse profile no painel.
4. O sistema carrega os dados do JSON.
5. O painel preenche automaticamente os campos.
6. Eu posso rodar:

   * validações individuais;
   * provisionamento granular;
   * setup completo a partir do profile.
7. O relatório final informa claramente qual profile foi usado.

O modo manual atual deve continuar funcionando exatamente como antes.

## SCHEMA DO SITE PROFILE JSON

O arquivo JSON deve seguir esta estrutura base:

```json
{
  "profile": {
    "slug": "aeconomia",
    "version": "1.0.0",
    "description": "Profile de configuração do portal A Economia"
  },
  "portal": {
    "name": "A Economia",
    "domain": "https://aeconomia.online",
    "niche": "financas",
    "language": "pt-BR",
    "timezone": "America/Sao_Paulo"
  },
  "wordpress": {
    "url": "https://aeconomia.online",
    "admin_user": "usuario_wp",
    "application_password": "",
    "wp_path": "/home/site/htdocs/aeconomia.online",
    "wp_cli_path": "/usr/local/bin/wp"
  },
  "ssh": {
    "host": "000.000.000.000",
    "port": 22,
    "user": "usuario_ssh",
    "auth_method": "password",
    "password": "",
    "key_path": ""
  },
  "seo": {
    "site_title": "A Economia",
    "tagline": "Notícias sobre economia, finanças e mercado",
    "permalink_structure": "/%postname%/",
    "blog_public": true,
    "comments_enabled": false,
    "ping_status": false,
    "rank_math": true,
    "instant_indexing": true
  },
  "plugins": {
    "required": [
      "seo-by-rank-math",
      "instant-indexing",
      "classic-editor"
    ],
    "optional": [
      "redis-cache",
      "site-kit-by-google"
    ],
    "skip": []
  },
  "content": {
    "create_pages": true,
    "create_categories": true,
    "create_test_post": true,
    "homepage_slug": "inicio"
  },
  "report": {
    "generate_markdown": true,
    "include_manual_pending_tasks": true
  }
}
```

## REGRAS DO SCHEMA

### Campos obrigatórios

Validar como obrigatórios:

* `profile.slug`
* `portal.name`
* `portal.domain`
* `portal.niche`
* `wordpress.url`
* `wordpress.admin_user`
* `wordpress.wp_path`
* `wordpress.wp_cli_path`
* `ssh.host`
* `ssh.port`
* `ssh.user`
* `ssh.auth_method`
* `seo.site_title`
* `seo.permalink_structure`

### Credenciais

Os campos abaixo podem estar vazios no JSON, porque o usuário poderá preencher pelo painel ou `.env`:

* `wordpress.application_password`
* `ssh.password`
* `ssh.key_path`

Mas o sistema deve validar que, no momento da execução, existe credencial suficiente para:

* autenticar no SSH;
* autenticar na REST API, quando tarefas REST forem executadas.

### SSH auth_method

Aceitar apenas:

* `password`
* `key`

Se `auth_method` for `password`, exigir senha no profile, formulário ou `.env`.

Se `auth_method` for `key`, exigir `key_path` no profile, formulário ou `.env`.

### Tipos

Validar tipos básicos:

* `ssh.port` deve ser número inteiro.
* flags em `seo`, `content` e `report` devem ser booleanas.
* listas de plugins devem ser arrays.
* URLs devem começar com `http://` ou `https://`.

### Nicho

Validar se `portal.niche` existe em `config/niches.json`.

Se não existir, retornar erro amigável informando os nichos disponíveis.
