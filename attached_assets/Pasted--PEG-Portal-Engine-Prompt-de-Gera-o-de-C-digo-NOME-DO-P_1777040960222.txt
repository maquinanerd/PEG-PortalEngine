# PEG Portal Engine — Prompt de Geração de Código

---

## NOME DO PROJETO

PEG Portal Engine

---

## OBJETIVO

Criar um sistema local em Python para provisionar e configurar automaticamente
instalações WordPress já criadas manualmente no CloudPanel.

O sistema NÃO deve instalar WordPress.
O sistema deve transformar um WordPress recém-criado em um portal editorial pronto para uso.

O código será gerado no Replit, mas deve rodar localmente no VS Code (Windows).

---

## REQUISITOS GERAIS

- Código portátil (não usar recursos exclusivos do Replit)
- Sem banco de dados
- Usar arquivos JSON para configuração
- Código modular e organizado
- Tratamento de erros obrigatório em todas as funções
- Logs em tela e arquivo
- Relatório final em Markdown
- Nunca expor credenciais completas nos logs ou relatório
- Usar pathlib.Path para todos os caminhos locais (compatibilidade Windows)
- Usar strings com "/" diretamente para caminhos remotos no VPS (nunca os.path.join para paths remotos)

---

## STACK

- Python 3.12+
- Flask (painel local)
- Paramiko (SSH)
- Requests (REST API WordPress)
- python-dotenv (.env)
- Bootstrap 5 (CDN)

## CAMADA SSH (ssh_client.py)

### Autenticação

Suportar dois modos mutuamente exclusivos:
1. Senha: usar `ssh_password` se preenchido
2. Chave privada: usar `ssh_key_path` (caminho local do arquivo) se senha estiver vazia

Ao carregar chave privada, usar `paramiko.RSAKey.from_private_key_file(path)`.
Tratar `paramiko.ssh_exception.PasswordRequiredException` se a chave for protegida por passphrase.

### Funções obrigatórias

```python
def conectar(host, port, user, password=None, key_path=None, timeout=30) -> paramiko.SSHClient
def executar(client, comando, timeout=60) -> dict  # {"stdout": str, "stderr": str, "exit_code": int}
def fechar(client)
def testar_conexao(host, port, user, password=None, key_path=None) -> dict  # {"ok": bool, "msg": str}
```

### Comportamento

- Timeout configurável por comando
- Retornar sempre stdout, stderr e exit_code
- Nunca lançar exceção não tratada: capturar e retornar erro no dict
- Logar cada comando executado (sem logar senhas)
- Retry automático de conexão: 3 tentativas com intervalo de 5 segundos

## CAMADA WP-CLI (wpcli.py)

### Regras obrigatórias para TODOS os comandos WP-CLI

1. **Sempre usar o caminho absoluto do binário** configurado em `WPCLI_BIN`
   (não assumir que `wp` está no PATH do shell SSH)

2. **Sempre incluir `--path={wp_path}`** onde `wp_path` é o caminho do WordPress no VPS

3. **Detectar automaticamente se SSH user == "root"** e, se sim, adicionar `--allow-root` em todos os comandos

4. **Usar `--format=json`** em todos os comandos que retornam dados (listas, opções, versões)
   Parsear com `json.loads()`. Tratar exceção `json.JSONDecodeError` se retorno não for JSON válido.

5. **Adicionar `--skip-themes --skip-plugins`** apenas em comandos de verificação/info
   para acelerar a execução (não usar em install/activate)

### Exemplo de comando correto

```python
# Internamente, os comandos devem ser montados assim:
def _build_cmd(self, subcommand: str, flags: list = None, json_output: bool = False) -> str:
    parts = [self.wpcli_bin, subcommand, f"--path={self.wp_path}"]
    if self.allow_root:
        parts.append("--allow-root")
    if json_output:
        parts.append("--format=json")
    if flags:
        parts.extend(flags)
    return " ".join(parts)
```

### Funções obrigatórias

```python
def verificar_wp() -> dict          # versão WP, URL, status
def verificar_wpcli() -> dict       # versão WP-CLI
def instalar_plugin(slug) -> dict   # instalar sem ativar
def ativar_plugin(slug) -> dict     # ativar plugin já instalado
def instalar_e_ativar(slug) -> dict # instalar + ativar em sequência
def listar_plugins_ativos() -> list # retorna lista de slugs ativos
def atualizar_opcao(key, value) -> dict
def configurar_permalink(estrutura) -> dict
def flush_rewrite() -> dict
def flush_cache() -> dict
def verificar_redis() -> dict       # executa "redis-cli ping" via SSH e verifica resposta PONG
```

## PLUGINS (config/plugins.json)

Definir no JSON com metadados:

```json
[
  {"slug": "seo-by-rank-math",    "nome": "Rank Math SEO",         "obrigatorio": true,  "requer_config_manual": true},
  {"slug": "fast-indexing-api",   "nome": "Instant Indexing",      "obrigatorio": true,  "requer_config_manual": false},
  {"slug": "classic-editor",      "nome": "Classic Editor",        "obrigatorio": true,  "requer_config_manual": false},
  {"slug": "redirection",         "nome": "Redirection",           "obrigatorio": true,  "requer_config_manual": false},
  {"slug": "wp-super-cache",      "nome": "WP Super Cache",        "obrigatorio": true,  "requer_config_manual": false},
  {"slug": "redis-cache",         "nome": "Redis Object Cache",    "obrigatorio": false, "requer_redis": true, "requer_config_manual": false},
  {"slug": "wp-mail-smtp",        "nome": "WP Mail SMTP",          "obrigatorio": true,  "requer_config_manual": true},
  {"slug": "site-kit-by-google",  "nome": "Site Kit by Google",    "obrigatorio": false, "requer_config_manual": true}
]
```

### Lógica de instalação

- Instalar e ativar todos os plugins com `"obrigatorio": true`
- Para `redis-cache`: antes de instalar, verificar se Redis está disponível no servidor
  rodando `redis-cli ping` via SSH. Instalar apenas se retornar `PONG`. Caso contrário,
  registrar no relatório como "Redis não disponível — redis-cache não instalado".
- Plugins com `"requer_config_manual": true` devem ser instalados/ativados normalmente,
  mas registrados no relatório como **pendências de configuração manual**.
- Plugins opcionais (`"obrigatorio": false`) devem ser pulados por padrão,
  com opção de ativar individualmente via checkbox no formulário.
- Para cada plugin: logar tentativa, resultado e erro (se houver).
  Não interromper o processo se um plugin falhar — registrar o erro e continuar.

- Atualizar `show_on_front` para `page`

**não pode ser automatizado** — registrar como pendência obrigatória no relatório.

## REST API (wp_rest.py)

### Autenticação

Usar HTTP Basic Auth com Application Password.

**Validação do Application Password antes de usar:**
- Remover espaços em excesso
- Verificar se tem exatamente 24 caracteres alfanuméricos (após remover espaços)
- Se inválido, retornar erro claro antes de fazer qualquer requisição

### Endpoints de teste

```
GET {wp_url}/wp-json/
GET {wp_url}/wp-json/wp/v2/users/me
```

### Funções obrigatórias

```python
def testar_api() -> dict

```

### Deduplicação obrigatória

Antes de criar qualquer página ou categoria:
1. Buscar itens existentes via GET
2. Comparar por slug
3. Pular criação e logar "já existe" se encontrar duplicata

## SETUP COMPLETO (tasks.py)

Executar em sequência, parando apenas em erros críticos (SSH e WP inválidos).
Erros não críticos (ex: um plugin falhou) devem ser logados e o processo continua.

```
Etapa 1:  Testar conexão SSH
Etapa 2:  Validar instalação WordPress (via WP-CLI)
Etapa 3:  Validar WP-CLI (versão e funcionamento)
Etapa 4:  Verificar disponibilidade do Redis (via SSH)
Etapa 5:  Instalar e ativar plugins obrigatórios
Etapa 6:  Instalar redis-cache (somente se Redis disponível)
Etapa 7:  Configurar opções do WordPress (via WP-CLI)
Etapa 8:  Configurar SEO técnico (permalink, indexação, homepage)
Etapa 9:  Testar REST API
Etapa 10: Criar categorias do nicho
Etapa 11: Criar páginas institucionais
Etapa 12: Criar conteúdo inicial (post rascunho)
Etapa 13: Flush rewrite rules e cache
Etapa 14: Gerar relatório final
```

Cada etapa deve retornar: `{"etapa": int, "nome": str, "status": "ok"|"erro"|"aviso", "detalhes": str}`

## LOGGER (logger.py)

- Usar módulo `logging` padrão do Python
- Nível configurável via `.env` (padrão: INFO)
- Dois handlers: console (stdout) e arquivo (`logs/peg_{timestamp}.log`)
- Formato: `[YYYY-MM-DD HH:MM:SS] [NIVEL] mensagem`
- Nunca logar: senhas, application passwords, chaves privadas
- Função helper: `log_credencial_segura(valor)` → retorna `"****"` sempre

## README

Incluir instruções:

1. Pré-requisitos: Python 3.12+, WP-CLI instalado no VPS
2. Criar virtualenv: `python -m venv venv`
3. Ativar venv: `venv\Scripts\activate` (Windows) ou `source venv/bin/activate` (Linux/Mac)
4. Instalar dependências: `pip install -r requirements.txt`
5. Copiar `.env.example` para `.env` e preencher
6. Rodar: `python app.py`
7. Acessar: http://127.0.0.1:5000
8. Nota sobre Application Password: gerar em WordPress → Usuários → Perfil → Senhas de aplicativo
